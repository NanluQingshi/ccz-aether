## 规则：每次代码变更完成后，主动给出 commit 注释

每当完成一个功能开发、UI 优化、重构或任何代码变更后，**无需用户提问**，在回复末尾直接给出建议的 commit message。

### Commit message 格式

```
<type>: <简洁描述>
```

### type 类型

| type | 适用场景 |
|------|---------|
| `feat` | 新功能、新页面、新接口 |
| `fix` | Bug 修复 |
| `refactor` | 重构，不改变外部行为 |
| `style` | 纯 UI / 样式调整，不涉及逻辑 |
| `chore` | 配置变更、依赖更新、脚本调整 |
| `docs` | 文档变更 |

### 规则

- 描述用**中文**，简洁，不超过 30 字
- 只描述"做了什么"，不解释"为什么"（why 属于 PR description）
- 涉及多个独立变更时，用 `、` 分隔写在同一行，或拆成多条供用户选择
- **不需要用户主动询问**，每次变更完成后自动附在回复结尾

### 示例

```
feat: 随想录新增类型筛选过滤功能
fix: token 过期后自动登出并弹出提示
style: Roadmap 页面标题栏布局优化
refactor: RoadmapPage 改为从 API 动态加载数据
chore: JWT 过期时间调整为 30 天
```
