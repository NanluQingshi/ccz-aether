## 规则：每次修复 Bug 后，将记录写入 bug 文档

每当完成一个 Bug 修复后，**自动**将本次 Bug 的背景、原因和解决方案追加写入项目的 Bug 记录文档。

### 文档位置

按问题所属模块追加到对应文件，不要新建单独的 bugs.md：

```
docs/troubleshooting/
├── 01-环境与构建.md     # 环境配置、构建、启动报错
├── 02-数据库.md         # SQL、MyBatis、数据迁移
├── 03-后端开发.md       # Spring Boot、安全、业务逻辑
├── 04-前端开发.md       # React、组件、状态管理
└── 05-网络与接口.md     # CORS、HTTP 状态码、接口联调
```

### 记录格式

追加到对应文件末尾，标题序号在该文件现有条目基础上递增：

```markdown
## N. 简短标题

**现象**：用户看到了什么问题，或触发条件是什么

**根因**：技术层面的真实原因

**解决**：改了哪个文件的哪个逻辑，可附关键代码片段
```

### 规则

- 按问题性质选择文件，跨模块问题放在影响最直接的那个文件
- **根因** 必须写清楚技术本质，不能只写"代码有问题"
- **解决** 要能让人日后快速理解改动意图，可附关键代码片段
- 每次修复后**无需用户提醒**，自动执行写入
