# 网络与接口问题

## 1. 登录请求被 CORS 拦截

**错误信息**
```
Access to XMLHttpRequest at 'http://localhost:8080/api/auth/login' from origin
'http://localhost:5173' has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

**原因**：浏览器在发送跨域请求前会先发一个 `OPTIONS` preflight 请求。Spring Security 的过滤器链在 Spring MVC 之前执行，`OPTIONS` 请求在到达 `WebMvcConfig` 的 CORS 配置之前就被 Security 拦截返回 401，响应头中没有 `Access-Control-Allow-Origin`，浏览器因此报跨域错误。

仅在 `WebMvcConfig.addCorsMappings()` 中配置 CORS 对 Security 无效，必须让 Security 层直接感知 CORS 配置。

**解决**：

1. 在 `SecurityConfig` 中通过 `.cors()` 注册 `CorsConfigurationSource` Bean：
```java
.cors(cors -> cors.configurationSource(corsConfigurationSource()))
```

2. 显式放行所有 `OPTIONS` 请求：
```java
.requestMatchers(HttpMethod.OPTIONS, "/**").permitAll()
```

3. 在 `SecurityConfig` 中定义 `corsConfigurationSource()` Bean，并删除 `WebMvcConfig` 中重复的 `addCorsMappings()`，避免两套配置冲突：
```java
@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(Arrays.asList(allowedOrigins));
    config.setAllowedMethods(List.of("GET","POST","PUT","PATCH","DELETE","OPTIONS"));
    config.setAllowedHeaders(List.of("*"));
    config.setAllowCredentials(true);
    config.setMaxAge(3600L);
    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/api/**", config);
    return source;
}
```

**原则**：使用 Spring Security 时，CORS 配置统一在 Security 层管理，不要同时使用 `WebMvcConfigurer.addCorsMappings()`。

## 2. 写接口持续返回 403，token 看起来有效

**现象**：前端请求 POST /api/musings、POST /api/books 等写接口，Network 面板显示 403，Request Headers 里 `Authorization: Bearer xxx` 是存在的，重启后端、清 localStorage、重新登录后问题依然存在。

**排查过程**：

在 `JwtAuthFilter` 加日志确认 `token_valid=false`；在 `JwtUtil.isTokenValid` 的 catch 块加日志输出异常信息，发现：
```
JWT expired 96317768 milliseconds ago at 2026-05-15T11:53:35.000Z.
Current time: 2026-05-16T14:38:52.768Z.
```

**根本原因：Zustand store 持有旧 token 未更新。**

流程如下：
1. 用户昨天登录，token（24小时有效）写入 localStorage 和 Zustand store 内存
2. 今天 token 过期，`isTokenValid` 返回 false，`JwtAuthFilter` 不设置 `SecurityContext`，请求到 `anyRequest().authenticated()` 被拒，返回 403
3. 用户清空 localStorage 后**未刷新页面**直接重新登录
4. 登录成功后新 token 写入 localStorage，但 `client.ts` 的 interceptor 读的是 `useAuthStore.getState().token`——而 Zustand store 在页面初始化时已将旧 token 加载到内存，没有刷新页面则内存值不变
5. 后续请求仍然带着内存里的旧 token，依然 403

**解决**：
- 清 localStorage 后**必须刷新页面**再重新登录，让 Zustand store 重新从 localStorage 初始化
- 将 JWT 过期时间从 24 小时延长到 7 天，减少频繁过期的情况：
```yaml
jwt:
  expiration-ms: 604800000   # 7天，原为 86400000（24小时）
```

**排查用临时日志**（问题解决后可移除）：
```java
// JwtAuthFilter
log.debug("[JWT] {} {} token_valid={}", request.getMethod(), request.getRequestURI(), valid);

// JwtUtil.isTokenValid catch 块
log.warn("[JWT] token invalid: {}", e.getMessage());
```
