# 后端开发问题

## 1. MyBatis-Plus Mapper 泛型类型推断失败

**错误信息**
```
BeanDefinitionStoreException: Invalid bean definition with name 'categoryMapper'
Invalid value type for attribute 'factoryBeanObjectType': java.lang.String
```

**原因**：MyBatis-Plus 通用版 starter（`mybatis-plus-boot-starter`）与 Spring Boot 3.3 存在兼容性问题，Spring 在初始化时对 `BaseMapper<T>` 泛型做类型推断，触发了已知 Bug。

**解决**：将 `pom.xml` 中的依赖替换为 Spring Boot 3 专用版：
```xml
<!-- 替换前 -->
<artifactId>mybatis-plus-boot-starter</artifactId>

<!-- 替换后 -->
<artifactId>mybatis-plus-spring-boot3-starter</artifactId>
```

## 2. PaginationInnerInterceptor 类找不到

**错误信息**
```
cannot find symbol: class PaginationInnerInterceptor
location: package com.baomidou.mybatisplus.extension.plugins.inner
```

**原因**：MyBatis-Plus 3.5.9 将分页插件从核心包拆分到独立模块 `mybatis-plus-jsqlparser`，原来的 `mybatis-plus-extension` 中不再包含。

**解决**：在 `pom.xml` 中额外引入：
```xml
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-jsqlparser</artifactId>
    <version>3.5.9</version>
</dependency>
```
包路径不变，`MybatisPlusConfig` 中的代码无需修改。

## 3. @Select 注解返回自定义 VO 导致 Bean 解析失败

**原因**：Spring 解析 Mapper Bean 时，对带有非实体返回类型的 `@Select` 方法存在边界问题，即使已换用 `spring-boot3-starter` 仍偶发。

**解决**：将 SQL 从 `@Select` 注解移入 XML Mapper 文件，接口只保留方法签名：
```java
// Mapper 接口：去掉 @Select，只保留签名
List<CategoryVO> selectWithPostCount();
```
在 `resources/mapper/CategoryMapper.xml` 中编写 SQL，并确保 `application.yml` 配置了扫描路径：
```yaml
mybatis-plus:
  mapper-locations: classpath:mapper/*.xml
```

## 4. 接口返回 500 但后端无任何日志输出

**现象**：前端请求返回 500，后端终端没有任何报错日志。

**原因**：`GlobalExceptionHandler` 捕获了所有异常但只构造了响应体，没有调用日志输出，异常被静默吞掉。

**解决**：在 `handleException` 中加上日志：
```java
@ExceptionHandler(Exception.class)
@ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
public ApiResponse<Void> handleException(Exception e) {
    log.error("Unhandled exception", e);
    return ApiResponse.error(500, "服务器内部错误: " + e.getMessage());
}
```

## 5. 仪表盘 getStats 接口 500：PostTag 无主键导致 selectCount 失败

**错误信息**
```
org.mybatis.spring.MyBatisSystemException: nested exception is ...
```

**原因**：`PostTag` 实体缺少 `@TableId` 注解，MyBatis-Plus 的 `selectCount` 在无主键实体上执行会报错。此外原代码用 `postTagMapper.selectCount()` 统计标签数，逻辑本身也错误——应统计 `tag` 表行数而非关联表。

**解决**：将 `PostService.getStats()` 中的统计改用 `tagMapper`：
```java
stats.setTotalTags(tagMapper.selectCount(null));
```

## 6. MyBatis-Plus insert() 后时间字段为 null

**现象**：调用 `mapper.insert(entity)` 后，返回的实体对象中 `createdAt`、`updatedAt` 字段为 null，尽管数据库已通过 `DEFAULT CURRENT_TIMESTAMP` 正确写入了时间值。

**原因**：`insert()` 只是执行 SQL 插入，不会回查数据库——数据库自动填充的字段值不会反映到 Java 对象上。

**解决**：使用 MyBatis-Plus 的自动填充机制，让 MP 在 Java 层主动填充时间，而非依赖数据库回写：

1. 新建 `MetaObjectHandlerConfig` 实现 `MetaObjectHandler`：
```java
@Component
public class MetaObjectHandlerConfig implements MetaObjectHandler {
    @Override
    public void insertFill(MetaObject metaObject) {
        LocalDateTime now = LocalDateTime.now();
        strictInsertFill(metaObject, "createdAt", LocalDateTime.class, now);
        strictInsertFill(metaObject, "updatedAt", LocalDateTime.class, now);
    }
    @Override
    public void updateFill(MetaObject metaObject) {
        strictUpdateFill(metaObject, "updatedAt", LocalDateTime.class, LocalDateTime.now());
    }
}
```

2. 在所有实体的时间字段上加注解：
```java
@TableField(fill = FieldFill.INSERT)
private LocalDateTime createdAt;

@TableField(fill = FieldFill.INSERT_UPDATE)
private LocalDateTime updatedAt;
```

**注意**：`strictInsertFill` 只在字段值为 null 时才填充，不会覆盖业务代码中手动赋值的字段（如 `Post.publishedAt`）。

## 7. 新增 Controller 后 GET 接口返回 403

**现象**：新建 `PracticeController` 并在 `SecurityConfig` 中添加 `permitAll()` 规则，但 `GET /api/practice` 仍返回 HTTP 403。

**根因**：Spring Boot 以 `mvn spring-boot:run` 启动后，JVM 加载的是 `target/classes/` 下已编译的字节码。源码文件（新实体类、Controller、修改后的 SecurityConfig）尚未重新编译，运行中的进程不感知任何源码变更，导致 SecurityConfig 中的新 `permitAll` 规则始终未生效。

**解决**：kill 旧进程后重新执行 `cd backend && mvn spring-boot:run`，Maven 会触发完整的 `compile` 阶段，所有新类和配置变更随之生效。
