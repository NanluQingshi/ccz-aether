# 环境与构建问题

## 1. Maven 找不到 spring-boot 插件

**错误信息**
```
No plugin found for prefix 'spring-boot' in the current project
```

**原因**：在错误目录执行命令，当前目录下没有 `pom.xml`，Maven 无法识别项目。

**解决**：在 `backend/` 目录下执行：
```bash
cd backend
mvn spring-boot:run
```

## 2. JDK 版本冲突导致编译失败

**错误信息**
```
Fatal error compiling: java.lang.ExceptionInInitializerError:
com.sun.tools.javac.code.TypeTag :: UNKNOWN
```

**原因**：Maven 使用了系统默认的 JDK 25（Homebrew 安装），与项目 `pom.xml` 要求的 Java 17 不兼容。`JAVA_HOME` 设置不带 `export` 时不会被子进程继承，Maven 仍会使用系统默认 JDK。

**解决**：在 `~/.zshrc` 中显式导出 JDK 17 路径：
```bash
export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-17.jdk/Contents/Home
export PATH="$JAVA_HOME/bin:$PATH"
```
然后重新加载：
```bash
source ~/.zshrc
mvn -version  # 确认显示 Java version: 17.x.x
```

## 3. 端口被占用导致后端启动失败

**错误信息**
```
Web server failed to start. Port 8080 was already in use.
```

**原因**：上一次后端进程没有正常退出（如直接关闭终端），进程仍在后台占用端口。这不是崩溃，是正常的启动失败退出。

**解决**：
```bash
# 方法一：杀掉占用 8080 的进程（推荐）
lsof -ti :8080 | xargs kill

# 方法二：改用其他端口启动
mvn spring-boot:run -Dspring-boot.run.arguments=--server.port=8081
# 同时需修改前端 .env 中的 VITE_API_BASE_URL 为对应端口
```

## 4. create-vite 版本与 Node.js 不兼容

**错误信息**
```
SyntaxError: The requested module 'node:util' does not provide an export named 'styleText'
```

**原因**：`create-vite@9.x` 要求 Node.js `>=20.19.0` 或 `>=22.12.0`，本机 Node.js 为 v18，不满足要求。

**解决**：指定兼容旧版 Node.js 的 create-vite：
```bash
npm create vite@5 frontend -- --template react-ts
```

## 5. 前端项目使用 pnpm，不能用 npm 安装依赖

**错误信息**
```
npm error Cannot read properties of null (reading 'matches')
```

**原因**：项目用 pnpm 初始化（存在 `pnpm-lock.yaml`），`node_modules` 是 pnpm 的符号链接结构，npm 的依赖解析器无法处理。

**解决**：一律使用 pnpm 管理依赖：
```bash
pnpm add <package>      # 安装新依赖
pnpm install            # 还原所有依赖
```

## 6. 引入 Tailwind CSS 后 Vite 报错：@import must precede all other statements

**错误信息**
```
[vite:css] @import must precede all other statements (besides @charset or empty @layer)
3  |  @tailwind utilities;
4  |
5  |  @import './theme.css';
   |  ^^^^^^^^^^^^^^^^^^^^^^
```

**背景**：项目原有纯 CSS 变量体系（`globals.css` 中用 `@import` 引入 `theme.css`、`components.css` 等），为接入 shadcn/ui 组件库引入了 Tailwind CSS v3，在 `globals.css` 顶部添加了 `@tailwind` 指令：

```css
/* 错误写法 */
@tailwind base;
@tailwind components;
@tailwind utilities;

@import './theme.css';
@import './components.css';
@import 'highlight.js/styles/github-dark.css';
```

**根本原因**：CSS 规范（以及 PostCSS/Vite 对其的实现）要求 **`@import` 语句必须出现在所有其他 CSS 声明之前**（`@charset` 和空 `@layer` 除外）。

`@tailwind base/components/utilities` 是 PostCSS 插件指令，在编译时会被展开成大量实际 CSS 规则。虽然源码里它们写在 `@import` 前面，但从 CSS 解析器的角度来看，这些指令一旦被处理就变成了"非 import 语句"，之后的 `@import` 就违反了规范，导致 Vite 在处理时报错。

**具体流程**：
1. PostCSS 处理 `globals.css`
2. `@tailwind base` 展开为 Tailwind 的 normalize/reset 规则（大量 CSS）
3. 展开后，后续的 `@import './theme.css'` 出现在实际 CSS 内容之后
4. CSS 规范不允许 `@import` 出现在非 `@charset`/空 `@layer` 之后，报错

**解决**：将所有 `@import` 移到文件最顶部，`@tailwind` 指令放在 `@import` 之后：

```css
/* 正确写法 */
@import './theme.css';
@import './components.css';
@import 'highlight.js/styles/github-dark.css';

@tailwind base;
@tailwind components;
@tailwind utilities;
```

**延伸**：这是在已有 CSS 体系的项目中接入 Tailwind 时的常见坑。CSS 文件中的语句顺序规则：
1. `@charset`（如果有）必须是第一条
2. `@import` 必须紧随其后，不能被任何其他规则打断
3. 其他所有规则（包括 `@tailwind` 展开后的内容）在最后

## 7. pnpm 突然报错：No versions available for @pnpm/exe

**现象**：本机已安装 pnpm，运行 `pnpm -v` 或 `pnpm dev` 时报错：
```
[ERROR] No versions available for @pnpm/exe. The package may be unpublished.
```
`npm install -g pnpm` 同样失败，报 `No versions available for undefined`。

**根因**：Node.js 16.9+ 内置了 corepack 工具。`package.json` 中的 `"packageManager": "pnpm@x.x.x"` 字段一旦被 corepack 感知，corepack 会拦截所有 pnpm 命令，强制去 npm registry 下载该指定版本，**完全绕过本机已安装的 pnpm**。网络不稳定或 registry 访问受阻时下载失败，导致本来正常的 pnpm 也无法使用。

**解决**：关闭 corepack，让系统回到使用本机安装的 pnpm：
```bash
corepack disable
pnpm -v  # 验证恢复正常
```

**延伸**：corepack 通常由升级 Node.js 或运行某些工具时被自动 enable。`packageManager` 字段本身不删也不影响，关掉 corepack 后该字段就只是一个声明，不再触发任何拦截。
