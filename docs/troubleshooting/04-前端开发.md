# 前端开发问题

## 1. MDEditor 编辑区文字不可见 【未解决】

**现象**：`PostEditorPage` 的 MDEditor 编辑区左侧输入内容后看不到文字，右侧预览区显示正常。

**根本原因**：MDEditor 的实际输入层 `.w-md-editor-text-input` 使用 `-webkit-text-fill-color: transparent` 将原始文字设为透明，显示内容依靠下层的语法高亮层 `.w-md-editor-text-pre`。语法高亮颜色由 `--color-prettylights-*` 系列 CSS 变量控制，需通过 `data-color-mode="light"` 注入浅色模式变量值才能正常显示深色文字。

**已尝试**：
- 在外层 `div` 上加 `data-color-mode="light"` → 无效
- 在 `MDEditor` 组件 props 上加 `data-color-mode="light"` → 无效
- CSS 强制覆盖 `color` 和 `background` → 无效，因为文字本身是透明的，覆盖 color 不影响显示

**环境**：`@uiw/react-md-editor@4.1.0`，后台明亮主题（白色背景）

**待排查方向**：
- 查阅 `@uiw/react-md-editor` v4 的 changelog，确认 `data-color-mode` 的注入方式是否有变化
- 尝试通过 `previewOptions` 或组件内部主题 API 切换 light 模式
- 参考官方 GitHub Issues 中关于 light mode 的讨论

## 2. 仪表盘页面一直加载无内容

**现象**：进入仪表盘后页面持续转圈，没有内容出现，无任何错误提示。

**原因**：`DashboardPage` 请求数据时没有 `.catch()`，接口失败时 `stats` 状态永远是 `null`，页面一直停在 loading 状态，错误被静默忽略。

**解决**：加上错误处理，失败时显示提示：
```tsx
const [stats, setStats] = useState<Stats | null>(null);
const [error, setError] = useState(false);

useEffect(() => {
  adminGetStats()
    .then((r) => setStats(r.data))
    .catch(() => setError(true));
}, []);

if (error) return <div>加载失败，请刷新重试</div>;
if (!stats) return <LoadingSpinner fullPage />;
```

**延伸**：所有异步数据请求都应处理 catch，避免因接口异常导致页面永久 loading。

## 3. 新增数据后页面崩溃：Cannot read properties of null (reading 'slice')

**错误信息**
```
TypeError: Cannot read properties of null (reading 'slice')
    at groupByMonth (MusingPage.tsx:36)
```

**现象**：调用新增接口成功后，页面立即崩溃，刷新后数据正常显示。

**原因**：两个问题叠加导致：
1. 后端 `insert()` 后直接返回实体对象，`createdAt` 为 null（MyBatis-Plus 未配置自动填充）
2. 前端 `handleAdd` 将 `res.data` 直接 `setMusings(prev => [res.data, ...prev])` 插入列表，`groupByMonth` 对每条数据调用 `m.createdAt.slice(0, 7)`，null 值导致崩溃

**解决**：

前端双重修复：

① `groupByMonth` 加防御性判断，跳过 `createdAt` 为 null 的数据：
```ts
for (const m of musings) {
  if (!m.createdAt) continue;  // 防御 null
  const key = m.createdAt.slice(0, 7);
  ...
}
```

② `handleAdd` 成功后改为重新拉取列表，而非手动插入可能不完整的响应数据：
```ts
await createMusing(req);
setInputVal('');
load();  // 重新拉取，数据以服务端为准
```

后端同步修复：参见「后端开发问题 #6」，配置 MyBatis-Plus 自动填充，从根源解决时间字段为 null 的问题。

**经验**：接口返回的数据不一定完整（时间字段、自增 ID 等可能缺失），新增后重新拉取列表比手动插入响应数据更可靠。

## 4. 封面图 URL 填入后图片不显示

**现象**：填入百度图片 CDN 链接（`bkimg.cdn.bcebos.com`），页面上封面图位置空白或显示破图。

**原因**：百度、知乎、微博等平台的图片 CDN 设置了**防盗链**（Hotlink Protection），服务器检测到请求的 `Referer` 不是本平台域名时，直接拒绝返回图片，浏览器收到 403 或空响应，`<img>` 无法渲染。

**解决**：

使用允许外链的图片来源：
- **豆瓣**：在豆瓣书目页面右键封面图 → 复制图片地址，格式为 `https://img1.doubanio.com/view/subject/l/public/s...jpg`
- **OpenLibrary**：`https://covers.openlibrary.org/b/isbn/{ISBN}-L.jpg`，完全开放，按 ISBN 拼接即可

同时在 `<img>` 标签加 `onError` 降级处理，图片加载失败时自动显示占位符，避免破图：
```tsx
<img
  src={book.cover}
  alt={book.title}
  className="book-cover-img"
  onError={() => setImgError(true)}
/>
// imgError 为 true 时渲染占位符
```

## 5. 操作失败 toast 只显示固定文案，不展示后端错误原因

**现象**：所有操作失败时只弹出"操作失败"、"删除失败"等硬编码文案，用户无法得知具体失败原因。

**根因**：各页面 `catch` 块写为 `catch {`（无参数），丢弃了 error 对象；即使有参数也没有从 `error.response.data.message` 中提取后端返回的错误信息。

**解决**：
1. 在 `client.ts` 响应拦截器中将 `error.response.data.message` 提升到 `error.message`
2. 封装 `getErrorMessage(e, fallback)` 工具函数，检测 `error.handled` 标志时返回 `null` 避免重复 toast
3. 各页面 catch 统一改为 `catch (e: unknown)` 并使用 `getErrorMessage`

## 6. Token 过期后触发两个错误提示弹窗

**现象**：Token 过期后页面同时弹出两个 toast，一个"登录已过期，请重新登录"，一个"操作失败"。

**根因**：拦截器在 401 时调用了一次 `addToast`，同时 `reject(error)` 后页面 catch 块读取 `e.message`（此时已被设为后端 message）再次 `addToast`，触发两次。

**解决**：拦截器处理 401 时在 error 上打标记 `error.handled = true`，`getErrorMessage` 检测到该标记返回 `null`，页面 catch 收到 `null` 则跳过 toast。

## 7. PostEditorPage 编辑模式下文章内容始终为空

**现象**：进入文章编辑页面时，Markdown 编辑器内容为空，无法看到原有的文章正文。

**根因**：代码中调用的是 `adminGetPosts(1, 999)` 返回 `PostVO[]`，而 `PostVO` 不含 `content` 字段（只有 `PostDetailVO extends PostVO` 才带 `content`）。导致 `setContent` 永远设置为 `undefined ?? ''`，编辑器始终空白。

**解决**：后端新增 `GET /api/admin/posts/{id}` 端点，返回 `PostDetailVO`（含 `content`）；前端新增 `adminGetPost(id)` API，编辑器 useEffect 改为调用该接口并执行 `setContent(post.content ?? '')`。

## 8. 书页间页面报错：thisYear is not defined

**现象**：访问 `/bookshelf` 页面时 ErrorBoundary 捕获到 `ReferenceError: thisYear is not defined`，页面白屏。

**根因**：`thisYear` 变量定义在 `doneThisYear` 的 `useMemo` 回调内部，但在 JSX 的 `{thisYear} 年读完` 处直接引用，超出了变量作用域。

**解决**：将 `thisYear` 提升到组件级别定义（`const thisYear = new Date().getFullYear().toString()`），`doneThisYear` 的 useMemo 使用外部变量并将其加入依赖数组。
