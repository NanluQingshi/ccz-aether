# 优化建议清单

> 基于 2026-05-21 全项目代码审查，覆盖安全、前端代码质量、UI/交互、性能、后端代码质量、工程化六个维度。  
> 优先级：🔴 立即 → 🟠 近期 → 🟡 中期 → 🟢 长期

---

## 一、安全

### 01 · Token 存储在 localStorage（XSS 风险）🔴
**位置**：`frontend/src/store/authStore.ts`

JWT 存于 `localStorage`，页面若存在任何 XSS 漏洞即可被脚本读取盗用。

**建议**：改为服务端通过 `Set-Cookie: HttpOnly; Secure; SameSite=Strict` 下发，前端不直接持有 Token 字符串。若保留 localStorage 方案，至少加内容安全策略（CSP）头。

---

### 02 · schema.sql 明文写死默认密码 🔴
**位置**：`backend/src/main/resources/schema.sql`

初始化脚本中直接写 `Admin@123456`，推送 Git 后永久留存于历史记录，任何拿到仓库的人都知道默认凭据。

**建议**：删除 SQL 中的硬编码密码，改为首次部署时通过环境变量注入或强制修改脚本生成随机密码。

---

### 03 · CORS 仅配置本地开发地址 🟠
**位置**：`backend/src/main/java/com/personalsite/blog/config/WebMvcConfig.java`

`allowedOrigins` 只含 `localhost:5173 / localhost:4173`，生产域名未加入白名单，上线后跨域请求直接失败。

**建议**：通过 `application-prod.yml` 配置生产域名，或使用环境变量 `ALLOWED_ORIGINS` 动态注入。

---

### 04 · JWT 有效期 30 天过长 🟡
**位置**：`backend/src/main/resources/application.yml` → `JWT_EXPIRATION_MS=2592000000`

单管理员场景下若 Token 泄露，攻击者有长达一个月的窗口期。

**建议**：缩短至 7 天，配合刷新 Token（Refresh Token）机制；同时实现登出时将 Token 加入黑名单（Redis Set）。

---

## 二、前端代码质量

### 01 · 组件卸载后仍尝试更新状态 🟠
**位置**：`frontend/src/pages/BlogDetailPage.tsx`、`frontend/src/pages/admin/PostEditorPage.tsx`

组件卸载后网络请求回调仍调用 `setState`，会产生内存泄漏警告，极端情况下渲染错误数据。

**建议**：在 `useEffect` 返回 cleanup 函数，通过 `AbortController` 取消未完成请求：
```ts
useEffect(() => {
  const controller = new AbortController();
  fetchPost(slug, { signal: controller.signal }).then(...);
  return () => controller.abort();
}, [slug]);
```

---

### 02 · BlogListPage 三个初始请求串行执行 🟠
**位置**：`frontend/src/pages/BlogListPage.tsx` 初始化逻辑

`getTags()` → `getCategories()` → `getPosts()` 顺序发出，浪费约 2 个 RTT。

**建议**：改为 `Promise.all` 并行发起：
```ts
const [tags, categories, posts] = await Promise.all([getTags(), getCategories(), getPosts(params)]);
```

---

### 03 · PostEditorPage 用 adminGetPosts(1, 999) 回填编辑数据 🟠
**位置**：`frontend/src/pages/admin/PostEditorPage.tsx`

仅为了获取当前文章详情，却拉取全量文章列表（最多 999 条），浪费带宽。

**建议**：后端补充 `GET /api/admin/posts/:id` 单篇详情接口，前端直接按 ID 请求。

---

### 04 · 高频渲染组件缺少 React.memo 🟠
**位置**：`frontend/src/components/blog/PostCard.tsx`、`frontend/src/components/ui/Pagination.tsx`

父组件翻页、筛选时触发状态更新，列表中所有 `PostCard` 都会重新渲染，即使 props 未变。

**建议**：对纯展示组件包裹 `React.memo`：
```ts
export default React.memo(PostCard);
```

---

### 05 · ToastContainer 无数量上限与去重 🟡
**位置**：`frontend/src/components/ui/ToastContainer.tsx`、`frontend/src/store/uiStore.ts`

接口连续失败时 Toast 会无限叠加，覆盖整个页面。

**建议**：在 `uiStore.addToast` 中限制最大数量（如 5 条），并对内容相同的 Toast 做去重。

---

### 06 · categoryId 类型混用 number 与空字符串 🟡
**位置**：`frontend/src/pages/admin/PostEditorPage.tsx`

`categoryId: number | ''` 是常见反模式，类型守卫繁琐且易遗漏。

**建议**：改为 `categoryId: number | null`，"未选择"状态用 `null` 表达，提交前校验 `categoryId !== null`。

---

## 三、UI / 交互

> *本节条目续接 `docs/frontend/` 已有的 01-响应式设计.md，编号从 02 开始。*

### 02 · Admin 文章列表在移动端横向溢出 🟠
**位置**：`frontend/src/pages/admin/PostManagerPage.tsx`

`<table>` 无移动端适配，手机上需横向滚动且操作按钮难以点击。

**建议**：小屏下切换为卡片式列表，或对表格容器加 `overflow-x: auto` 并设置合理 `min-width`。

---

### 03 · Issue Board 固定高度在小屏幕失效 🟠
**位置**：`frontend/src/styles/components.css` → `.issue-board`

`height: calc(100vh - 302px)` 在视口高度低于 600px 的设备上高度趋近于零甚至为负。

**建议**：改为 `min-height: 400px` + `max-height: calc(100vh - 302px)`，或去掉固定高度改为自然流式布局。

---

### 04 · Footer 在短页面遮挡底部内容 🟡
**位置**：`frontend/src/styles/components.css` → `.footer`、`.main-content`

Footer `position: fixed; bottom: 0`，`main-content` 的 `padding-bottom` 有时不足以完全露出最后一行内容，中间屏幕尺寸（iPad 竖屏等）下更明显。

**建议**：精确计算 Footer 实际高度后统一赋值给 CSS 变量，`main-content` 引用该变量作为 `padding-bottom`。

---

### 05 · 状态徽章仅靠颜色区分（色盲不友好） 🟡
**位置**：`frontend/src/styles/components.css` → `.status-*`、`.priority-*`

`已发布/草稿`、`Todo/In Progress/Done`、优先级高低等状态仅用色彩区分，色盲用户无法辨别。

**建议**：在徽章内同时加图标（如 ✓ / ○ / ✗）或形状差异，颜色作为辅助而非唯一区分维度。

---

### 06 · ToastContainer 缺少无障碍属性 🟡
**位置**：`frontend/src/components/ui/ToastContainer.tsx`

通知区域没有 `role="status"` 或 `aria-live="polite"`，屏幕阅读器不会主动播报新通知内容。

**建议**：
```tsx
<div role="status" aria-live="polite" aria-atomic="false" className="toast-container">
```

---

### 07 · 分页组件功能单一，无跳页输入 🟡
**位置**：`frontend/src/components/ui/Pagination.tsx`

只有上一页/下一页按钮，文章超过 5 页时翻页体验很差，也缺少当前页的 `aria-current="page"` 标记。

**建议**：增加"跳转到第 N 页"输入框（blur 时触发），并在当前页按钮加 `aria-current="page"`。

---

### 08 · 编辑器无本地草稿自动保存 🟡
**位置**：`frontend/src/pages/admin/PostEditorPage.tsx`

浏览器意外关闭或标签页崩溃时，未保存的草稿内容直接丢失。

**建议**：每 30 秒或内容变化后将标题、正文写入 `localStorage`（key 含 `postId` 或 `draft`），进入编辑页时检测并提示恢复。

---

## 四、性能

### 01 · Markdown 编辑器未懒加载 🟠
**位置**：`frontend/src/pages/admin/PostEditorPage.tsx`

`@uiw/react-md-editor` 体积约 100 KB（gzip 后），但随主包一起在首屏加载，所有访客都要下载这段只有管理员才用到的代码。

**建议**：
```ts
const MDEditor = React.lazy(() => import('@uiw/react-md-editor'));
```
配合 `<Suspense fallback={<LoadingSpinner />}>` 包裹使用。

---

### 02 · Hero 区 12 个标签同时跑无限 CSS 动画 🟠
**位置**：`frontend/src/styles/components.css` → `.hero-tag`

12 个元素持续 `animation: hero-tag-float linear infinite`，在低端设备和低功耗模式下持续消耗合成线程。

**建议**：加 `prefers-reduced-motion` 媒体查询关闭动画；或改为只在元素可见时（`IntersectionObserver`）播放，不可见时 `animation-play-state: paused`。

---

### 03 · PostCard 封面图无懒加载 🟠
**位置**：`frontend/src/components/blog/PostCard.tsx`

列表页所有封面图同时发起请求，首屏流量大、可视区外图片白白占用带宽。

**建议**：图片标签加 `loading="lazy"` 属性（一行代码）；有条件时进一步提供 WebP 格式与 `srcset`。

---

### 04 · 大依赖包未拆分，打包进主 chunk 🟡
**位置**：`frontend/vite.config.ts`（当前未配置 `manualChunks`）

`highlight.js`、`recharts`、`react-markdown` 等大依赖与业务代码打在同一包中，导致主包体积偏大，首屏 JS 解析时间长。

**建议**：在 `vite.config.ts` 的 `build.rollupOptions.output.manualChunks` 中拆分：
```ts
manualChunks: {
  'vendor-md': ['@uiw/react-md-editor', 'react-markdown', 'remark-gfm'],
  'vendor-chart': ['recharts'],
  'vendor-highlight': ['highlight.js'],
}
```

---

### 05 · 博客详情页每次进入都全量重新请求 🟡
**位置**：`frontend/src/pages/BlogDetailPage.tsx`

文章内容几乎不变，但每次进入详情页都重新 fetch，无任何缓存策略。

**建议**：后端在响应头加 `Cache-Control: public, max-age=300, stale-while-revalidate=600`，或前端用 `useMemo` + 简单 Map 缓存已加载的 slug → 内容，避免重复请求。

---

## 五、后端代码质量

> *本节条目续接 `docs/troubleshooting/` 已有的 01-05，编号从 06 开始。*

### 06 · 删除文章操作缺少事务保护 🔴
**位置**：`backend/src/main/java/.../service/impl/PostServiceImpl.java` → `delete()`

先删 `post`，再删 `post_tag` 关联，若第二步失败则文章已删但标签关联脏数据残留，且无法回滚。

**建议**：方法加 `@Transactional`，确保两个删除操作原子性执行。

---

### 07 · Slug 重复时直接抛数据库唯一键异常 🟠
**位置**：`backend/src/main/java/.../service/impl/PostServiceImpl.java` → `create()` / `update()`

唯一键冲突时底层 `DataIntegrityViolationException` 未被拦截，透传给前端后显示为系统级错误信息，用户体验差。

**建议**：在 service 层 `INSERT` 前先 `SELECT COUNT(*)` 检查 slug 是否已存在，若重复抛 `BizException(ErrorCode.SLUG_DUPLICATE)`，返回友好提示。

---

### 08 · JWT secret 无最小长度校验 🟠
**位置**：`backend/src/main/resources/application.yml` → `jwt.secret`

注释提醒"生产环境请更换"但无代码层面强制。HMAC-SHA256 密钥短于 32 字节时安全性明显下降。

**建议**：在 `JwtUtil` 初始化（`@PostConstruct`）时断言 secret 长度 ≥ 32，不满足则直接抛异常阻止启动：
```java
@PostConstruct
void validate() {
    Assert.isTrue(secret.length() >= 32, "JWT secret must be at least 32 characters");
}
```

---

### 09 · schema.sql 已建全文索引但从未使用 🟡
**位置**：`backend/src/main/resources/schema.sql`、`PostMapper`

`post` 表有 `FULLTEXT(title, content)` 索引，但没有对应的搜索接口，前端也没有搜索入口，索引仅占空间没有收益。

**建议**：要么实现搜索功能（`MATCH(title, content) AGAINST(? IN BOOLEAN MODE)`），要么在确认短期不做搜索的情况下删除全文索引以减少写入开销。

---

### 10 · admin 文章列表接口是否修复了 N+1 🟡
**位置**：`backend/src/main/java/.../service/impl/PostServiceImpl.java` → `adminListAll()`

方案一已修复公开博客列表的 N+1，但需确认 admin 侧分页接口是否复用了同一优化后的 mapper，若单独实现则可能仍存在逐篇查标签的问题。

**建议**：核查 `adminListAll` 的 SQL，若未 JOIN tags 则参照方案一补充批量查询逻辑。

---

## 六、工程化

### 01 · 缺少 React Error Boundary 🟠
**位置**：`frontend/src/router/` 路由配置

任何子组件抛出未捕获的 JS 异常都会导致整页白屏，无任何友好提示。

**建议**：在路由层级包裹 `<ErrorBoundary>` 组件，捕获渲染错误后展示降级 UI（"页面出了点问题，请刷新"）并上报错误信息。

---

### 02 · 博客详情页缺少 SEO meta 标签 🟡
**位置**：`frontend/src/pages/BlogDetailPage.tsx`

`<title>` 和 `<meta name="description">` 全站固定，搜索引擎无法为各篇文章建立有效索引，Open Graph 分享预览也无文章信息。

**建议**：引入 `react-helmet-async`，在详情页动态注入：
```tsx
<Helmet>
  <title>{post.title} - 博客名</title>
  <meta name="description" content={post.summary ?? post.content.slice(0, 160)} />
  <meta property="og:title" content={post.title} />
  <meta property="og:image" content={post.coverImage} />
</Helmet>
```

---

### 03 · 后端无健康检查端点 🟡
**位置**：`backend/src/main/java/com/personalsite/blog/` 无 actuator 或自定义 health

Docker Compose / k8s 的 `healthcheck` 无探针可用，容器编排无法判断服务是否真正就绪。

**建议**：引入 `spring-boot-starter-actuator` 并配置只暴露 `/actuator/health`：
```yaml
management:
  endpoints:
    web:
      exposure:
        include: health
  endpoint:
    health:
      show-details: never
```

---

### 04 · 请求无关联 ID，前后端日志无法对应 🟢
**位置**：`backend/src/main/java/.../config/`（无 MDC 过滤器）、`frontend/src/api/client.ts`

线上排查问题时，前端看到的错误无法与后端日志关联，定位耗时长。

**建议**：
- 后端：在 `Filter` 中为每个请求生成 `X-Request-Id`（UUID），写入 MDC，并在响应头返回。
- 前端：`axios` 拦截器在请求头附带 `X-Request-Id`，Toast 显示错误时顺带展示该 ID，便于反馈。
