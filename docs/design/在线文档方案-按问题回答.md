# 在线文档方案（按问题回答）

## 1. 编辑器集成与保存机制

### 如何集成富文本编辑器？
本题中，前端接入富文本编辑器后，需要维护四类核心状态：当前编辑内容、保存状态、未提交变更、本地草稿。编辑器初始化时先调用文档详情接口获取最新内容和当前版本号，然后将内容加载到编辑器中。用户输入时，先更新本地编辑器状态，再把当前页面标记为“未保存”，同时启动自动保存计时器。若系统支持多人协同编辑，则本地变更还要通过 WebSocket 实时同步给其他在线用户。

可以维护如下状态：

- `editorState`：当前文档内容
- `saveState`：保存状态，可取 `未保存 / 保存中 / 已保存 / 保存失败 / 离线编辑中`
- `pendingChanges`：尚未提交到服务端的变更
- `baseVersion`：当前编辑所基于的版本号
- `localDraft`：浏览器本地草稿

### 自动保存和手动保存如何实现？
自动保存通过“防抖定时器”实现。用户停止输入 5 秒后触发一次保存请求；如果用户持续输入，也可以每隔一段时间做一次兜底保存。手动保存则由用户点击保存按钮后立即发起请求，不经过防抖等待。

两者流程一致，区别主要在触发方式和历史记录表现：

- 自动保存：用于降低丢稿风险
- 手动保存：用于形成明确版本节点

保存时，前端向服务端提交文档内容和当前 `baseVersion`。服务端保存成功后返回新的版本号，前端据此更新 `baseVersion`，并将状态切换为“已保存”。

### 如何展示保存状态？
保存状态直接展示在编辑器顶部或标题栏，用户能实时看到当前文档状态。状态可以分为：

- `未保存`：本地已有修改，但尚未提交
- `保存中...`：请求已经发出，等待服务端响应
- `已保存`：当前内容已经持久化
- `保存失败`：服务端返回失败或网络异常
- `离线编辑中`：当前无法连接服务端，只保存在本地

同时显示最近一次成功保存时间，例如“已保存于 14:32:10”，这样用户能够明确感知当前内容是否真正落库。

### 保存失败时如何处理？丢失数据时如何处理重试？
保存失败后，前端不能清空未提交变更，而是保留 `pendingChanges`，并将状态改为“保存失败”。系统可以执行自动重试，也可以允许用户手动点击“重试”。重试时仍然提交原来的未保存内容，直到服务端确认保存成功为止。

### 浏览器异常关闭后如何处理？
浏览器异常关闭前，未保存内容要先写入本地草稿，例如 `IndexedDB`。草稿中至少保存：文档 ID、当前内容、`baseVersion`、最后编辑时间。用户再次进入文档时，系统先检查是否存在本地草稿；如果存在，则提示用户是否恢复未保存内容。这样即使页面崩溃或浏览器被强制关闭，也不会直接丢失编辑结果。

### 保存接口示例

```http
POST /api/documents/{id}/save
```

请求体：

```json
{
  "baseVersion": 12,
  "content": {},
  "saveMode": "AUTO",
  "summary": ""
}
```

响应体：

```json
{
  "revisionId": 301,
  "version": 13,
  "savedAt": "2026-05-27T15:00:00Z"
}
```

## 2. 版本历史界面与回滚操作

### 如何获取并展示历史版本？
版本历史由服务端提供分页查询接口，前端在侧边栏或弹窗中展示。每条历史记录包含以下信息：版本号、保存时间、操作者、保存方式、版本说明。用户点击某条历史记录后，可以查看当时的文档内容。

接口如下：

```http
GET /api/documents/{id}/revisions?page=1&pageSize=20
```

若需要查看某一个具体版本的详细内容，则调用：

```http
GET /api/documents/{id}/revisions/{revisionId}
```

### 版本对比（diff）如何实现？
Diff 的实现方式是：取两个版本的内容，分别作为旧版本和新版本输入 Diff 算法，得到新增、删除、修改三类结果，再由前端高亮展示。

可以支持两种常见场景：

- 当前版本和某个历史版本对比
- 任意两个历史版本之间对比

接口如下：

```http
GET /api/documents/{id}/diff?fromVersion=10&toVersion=13
```

前端展示时，可以采用左右对比布局：

- 绿色表示新增
- 红色表示删除
- 黄色表示修改

### 回滚操作的流程是怎样的？
回滚不是直接覆盖旧版本，而是把目标历史版本的内容恢复为当前最新内容，并生成一个新的版本号。这样历史链条仍然完整，能够知道“这次回滚是由哪个版本恢复而来”。

流程如下：

1. 用户在历史列表中选择目标版本
2. 点击“回滚到此版本”
3. 系统检查当前文档是否存在未保存修改
4. 系统校验当前 `headVersion` 是否仍然等于用户发起操作时看到的版本
5. 校验通过后，把目标版本内容写入文档主表
6. 新增一条 `ROLLBACK` 类型的历史记录
7. 返回新的版本号，并通知在线协作者刷新文档内容

回滚接口如下：

```http
POST /api/documents/{id}/rollback
```

请求体：

```json
{
  "targetVersion": 8,
  "baseVersion": 13,
  "reason": "恢复到误删前版本"
}
```

### 回滚时如何处理冲突？
如果用户发起回滚时，当前文档已经被其他人保存过新内容，则说明 `headVersion` 已经变化，此时不能直接回滚，否则会覆盖他人的修改。系统应返回版本冲突错误，提示用户先刷新最新内容，再决定是否继续回滚。

### 当当前有未保存修改时如何处理？
如果当前编辑器中还有未保存修改，回滚前必须先处理这些内容。处理方式有三种：

- 先保存当前内容，再执行回滚
- 先把当前修改保存为本地草稿，再执行回滚
- 放弃当前未保存修改，直接回滚

在考试回答中，写出这三种处理分支即可，重点说明：回滚前必须先处理未保存内容，不能直接覆盖。

## 3. 文档存储与版本控制

### 设计文档及版本历史表结构
系统至少包含两张核心表：文档主表和版本历史表。

### 1）文档主表 `document`
文档主表保存当前最新内容，用于日常读取和编辑。

```sql
CREATE TABLE document (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(255) NOT NULL,
  content_json LONGTEXT NOT NULL,
  head_version BIGINT NOT NULL,
  last_saved_by BIGINT,
  last_saved_at DATETIME NOT NULL,
  deleted TINYINT NOT NULL DEFAULT 0,
  create_time DATETIME NOT NULL,
  update_time DATETIME NOT NULL
);
```

字段含义如下：

- `content_json`：当前最新文档内容
- `head_version`：当前最新版本号
- `last_saved_by`：最后一次保存人
- `last_saved_at`：最后保存时间

### 2）历史版本表 `document_revision`
历史版本表保存每一个可追溯版本。

```sql
CREATE TABLE document_revision (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  document_id BIGINT NOT NULL,
  version_no BIGINT NOT NULL,
  save_mode VARCHAR(20) NOT NULL,
  content_json LONGTEXT NOT NULL,
  diff_json LONGTEXT NULL,
  summary VARCHAR(500) NULL,
  operator_id BIGINT NOT NULL,
  rollback_from_revision_id BIGINT NULL,
  created_at DATETIME NOT NULL,
  UNIQUE KEY uk_doc_version (document_id, version_no)
);
```

字段含义如下：

- `version_no`：同一文档内递增版本号
- `save_mode`：保存方式，如 `AUTO`、`MANUAL`、`ROLLBACK`
- `content_json`：该版本对应的完整内容快照
- `diff_json`：该版本与上一版本的差异信息
- `rollback_from_revision_id`：若该版本由回滚产生，则记录来源版本

### 定义保存文档的 API
保存接口定义如下：

```http
POST /api/documents/{id}/save
```

请求体：

```json
{
  "baseVersion": 13,
  "content": {},
  "saveMode": "MANUAL",
  "summary": "完成需求说明部分"
}
```

该接口完成三个动作：

1. 校验当前版本是否冲突
2. 更新文档主表中的最新内容和最新版本号
3. 向历史版本表插入一条新记录

### 如何用版本号实现乐观锁并避免覆盖？
乐观锁的核心是：前端保存时必须带上自己编辑所基于的版本号 `baseVersion`。服务端收到请求后，先读取数据库中的 `head_version`，再和 `baseVersion` 比较。

- 如果两者相等，说明当前没有其他人抢先提交，可以保存
- 如果两者不相等，说明在本次提交前已经有人保存了新版本，此时返回冲突错误，拒绝覆盖

实现时可以直接使用带条件的更新语句：

```sql
UPDATE document
SET content_json = ?, head_version = head_version + 1, update_time = NOW()
WHERE id = ? AND head_version = ?;
```

根据影响行数判断结果：

- 影响 1 行：保存成功
- 影响 0 行：发生版本冲突

这样就通过版本号实现了乐观锁，避免了后提交覆盖先提交的问题。

### 版本历史是保存全量还是增量？
版本历史可以有两种保存方式：全量快照和增量记录。

- 全量快照：每次保存都记录完整文档内容，优点是回滚和恢复简单，缺点是占用空间更大
- 增量记录：每次只记录变化部分，优点是节省空间，缺点是恢复过程复杂，回滚时需要按顺序回放多个版本

在本题场景下，直接写“历史版本表保存完整快照，同时可附带保存 diff 信息”即可。这样表述既说明了存储结构，也说明了版本对比和回滚为何容易实现。
