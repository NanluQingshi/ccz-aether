# 在线文档方案

## 目标

面向多人实时协作编辑场景，设计一套支持以下能力的在线文档系统：

- 多人同时编辑同一文档
- 5 秒自动保存 + 手动保存
- 清晰的保存状态反馈
- 版本历史查看、Diff 对比、版本回滚
- 断网、保存失败、浏览器异常关闭后的数据恢复
- 通过版本号和乐观锁避免覆盖更新

本文基于当前项目技术栈进行设计：前端使用 React，后端使用 Spring Boot，主存储使用 MySQL；实时协作链路建议增加 WebSocket，必要时引入 Redis 做会话状态和广播扩展。

## 一、总体架构

### 1.1 分层设计

- 前端编辑器层：负责富文本编辑、保存状态展示、版本历史面板、Diff 预览、回滚交互
- 协同编辑层：负责多人实时同步、在线用户状态、操作广播、断线重连
- 文档服务层：负责保存、手动保存、版本生成、版本查询、回滚、权限校验
- 存储层：负责当前文档内容、历史版本、快照、操作日志、草稿恢复数据

### 1.2 推荐技术方案

- 编辑器：`TipTap / ProseMirror` 或 `Slate`
- 协同协议：优先 `Yjs + WebSocket`（CRDT），也可使用 OT 方案
- 后端实时通道：Spring WebSocket
- 数据库：MySQL
- 本地草稿缓存：浏览器 `IndexedDB`
- 服务端缓存/广播扩展：Redis（多实例部署时推荐）

### 1.3 为什么优先选 CRDT

对于“多人同时编辑同一文档”的核心诉求，CRDT 比“定时整文覆盖保存”更适合：

- 每个客户端产生本地操作，先本地应用，再异步同步
- 冲突由 CRDT 自动合并，减少“谁覆盖谁”的问题
- 弱网和短时离线恢复能力更好
- 与版本系统并不冲突：实时协作解决“当前内容怎么合”，版本系统解决“历史状态怎么查、怎么回”

## 二、编辑器集成与保存机制

## 2.1 编辑器接入方式

前端编辑器维护三份状态：

- `editorState`：当前可见内容状态
- `pendingOps`：尚未持久化到服务端的本地操作
- `saveState`：保存状态，取值建议为 `idle / dirty / saving / saved / error / offline`

用户在编辑器中输入时：

1. 先更新本地编辑器内容
2. 生成协同操作并通过 WebSocket 广播给其他协作者
3. 将本次变更加入 `pendingOps`
4. 将页面状态置为 `dirty`
5. 触发自动保存计时器

## 2.2 自动保存设计

建议采用“防抖 + 周期兜底 + 关键时机强制落盘”的组合方案。

### 自动保存触发规则

- 用户停止输入 5 秒后触发一次自动保存
- 即使持续输入，也每隔 30 秒做一次强制保存兜底
- 页面失焦、切路由、关闭页面前触发一次快速提交
- 网络恢复后立即重试未成功保存的数据

### 保存接口输入

自动保存请求建议至少包含：

```json
{
  "documentId": 123,
  "baseVersion": 57,
  "clientId": "web-user-01",
  "content": "序列化后的文档内容或快照",
  "ops": [],
  "saveMode": "AUTO"
}
```

其中：

- `baseVersion`：当前客户端认为的文档基线版本
- `content`：可选，适合直接保存整份文档快照
- `ops`：可选，适合保存增量操作
- `saveMode`：区分自动保存和手动保存，便于历史展示

### 保存策略建议

若系统初期希望实现简单稳定，可采用：

- 实时协作层用 CRDT 同步内容
- 持久化层按“整文快照 + 版本号”保存

这样实现成本较低，回滚和 Diff 也更清晰。后续再逐步增加增量操作日志。

## 2.3 手动保存设计

手动保存应与自动保存区分处理：

- 手动保存立即提交，不等待 5 秒防抖
- 手动保存成功后创建一个明确版本节点
- 可允许用户填写版本说明，如“完成第一版初稿”
- 历史列表中优先展示手动保存版本，自动保存版本可弱化显示

## 2.4 保存状态反馈

编辑器顶部或标题栏右侧建议展示统一保存状态：

- `未保存`：本地有变更，尚未提交
- `保存中...`：请求已发出，等待服务端确认
- `已保存`：当前内容已成功落库
- `保存失败`：请求失败，可点击重试
- `离线编辑中`：网络断开，仅保存在本地草稿

推荐交互细节：

- 文本提示：如“已保存于 14:32:10”
- 图标颜色区分状态
- `保存失败` 时提供“重试”按钮
- 离开页面前若仍为 `dirty` 或 `saving`，给出浏览器离开提醒

## 2.5 保存失败和异常恢复

### 失败重试

保存失败后不要立刻丢弃 `pendingOps`，而是：

- 将失败请求写入本地重试队列
- 前端指数退避重试，如 2s / 5s / 10s
- 用户手动点击“立即重试”时可直接发起一次

### 本地草稿恢复

浏览器侧使用 `IndexedDB` 持久化以下信息：

- 最近一次未保存内容
- `pendingOps`
- 文档 ID
- 对应 `baseVersion`
- 最后编辑时间

当浏览器崩溃、页面刷新或意外关闭后：

1. 打开文档时先读取本地草稿
2. 若发现存在未提交草稿，提示用户“恢复未保存内容”
3. 用户可选择恢复到编辑器，或放弃本地草稿

### 服务端确认机制

每次保存成功后，服务端返回：

```json
{
  "success": true,
  "version": 58,
  "revisionId": 901,
  "savedAt": "2026-05-27T14:32:10Z"
}
```

前端仅在收到确认后：

- 清空已提交的 `pendingOps`
- 将 `baseVersion` 更新为新版本号
- 将状态从 `saving` 切换到 `saved`

## 三、版本历史界面与回滚操作

## 3.1 历史列表展示

建议在编辑器右侧抽屉或独立侧栏展示版本历史。

每条版本记录展示：

- 版本号，如 `v58`
- 保存类型：自动保存 / 手动保存 / 回滚生成
- 保存时间
- 操作者
- 版本说明（手动保存时可填）
- 内容摘要或改动概览

### 列表示例

- `v58` 手动保存 14:32 张三 - 完成目录结构调整
- `v57` 自动保存 14:25 张三
- `v56` 回滚生成 13:48 李四 - 从 v52 恢复

## 3.2 Diff 对比实现

版本对比支持两种模式：

- 当前版本 vs 指定历史版本
- 任意两个历史版本之间对比

### 实现建议

如果文档是富文本 JSON 结构：

- 服务端可先将结构转换为标准文本块/段落块，再做文本 Diff
- 前端展示时按“新增 / 删除 / 修改”高亮

如果使用 ProseMirror/Tiptap：

- 可基于节点树做结构化 Diff
- 块级别展示更清晰，例如标题、段落、列表、表格分别标记改动

### UI 建议

- 左右对比视图：左侧旧版，右侧新版
- 行内高亮：绿色新增、红色删除、黄色修改
- 提供“仅看变化内容”开关

## 3.3 回滚流程设计

回滚不建议直接覆盖旧版本，而应采用“基于历史版本生成一个新版本”的方式。

推荐流程：

1. 用户选择某个历史版本，点击“回滚到此版本”
2. 系统先检查当前编辑器是否有未保存修改
3. 若有未保存修改，弹框提示处理方式
4. 用户确认后，服务端根据目标版本内容生成新的 head 版本
5. 新版本写入历史表，并记录 `rollbackFromRevisionId`
6. 广播给当前协作者刷新内容

### 为什么回滚要生成新版本

这样做有三个好处：

- 历史链完整，回滚行为可追踪
- 原目标版本仍然保留，不会被覆盖
- 再次回退时容易继续操作

## 3.4 回滚时有未保存修改的处理

这是回滚设计的关键点。建议提供三个选项：

- `先保存当前内容，再回滚`
- `将当前修改保存为本地草稿，再回滚`
- `放弃未保存修改，直接回滚`

默认推荐第二种：

- 风险最小
- 不会污染正式版本历史
- 用户仍可在需要时恢复草稿

若存在多人协作，回滚前还应校验当前 headVersion 是否已变化；若用户基于旧版本视图发起回滚，而期间他人又产生了新版本，应提示用户先刷新再确认。

## 四、文档存储与版本控制

## 4.1 表结构设计

建议至少包含以下三张核心表。

### 文档主表 `document`

用于快速读取当前最新内容。

```sql
CREATE TABLE document (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(255) NOT NULL,
  content_json LONGTEXT NOT NULL,
  head_version BIGINT NOT NULL,
  last_saved_by BIGINT,
  last_saved_at DATETIME NOT NULL,
  deleted TINYINT NOT NULL DEFAULT 0,
  create_time DATETIME NOT NULL,
  update_time DATETIME NOT NULL
);
```

字段说明：

- `content_json`：当前最新文档内容
- `head_version`：当前最新版本号

### 版本表 `document_revision`

记录每个历史版本。

```sql
CREATE TABLE document_revision (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  document_id BIGINT NOT NULL,
  version_no BIGINT NOT NULL,
  save_mode VARCHAR(20) NOT NULL,
  content_json LONGTEXT NOT NULL,
  diff_json LONGTEXT NULL,
  summary VARCHAR(500) NULL,
  operator_id BIGINT NOT NULL,
  rollback_from_revision_id BIGINT NULL,
  created_at DATETIME NOT NULL,
  UNIQUE KEY uk_doc_version (document_id, version_no)
);
```

字段说明：

- `version_no`：文档内单调递增版本号
- `save_mode`：`AUTO` / `MANUAL` / `ROLLBACK`
- `content_json`：该版本完整快照
- `diff_json`：可选，存储与上个版本的差异
- `rollback_from_revision_id`：若由回滚产生，用于追踪来源

### 草稿恢复表 `document_draft_recovery`（可选）

如果希望服务端也能托管恢复数据，可增加此表。

```sql
CREATE TABLE document_draft_recovery (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  document_id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  base_version BIGINT NOT NULL,
  content_json LONGTEXT NOT NULL,
  updated_at DATETIME NOT NULL,
  UNIQUE KEY uk_doc_user (document_id, user_id)
);
```

若暂时不做服务端草稿托管，也可以仅保存在前端 IndexedDB。

## 4.2 版本号设计

版本号使用单文档内递增整数即可，例如：`1, 2, 3, 4...`。

### 版本号用途

- 标识文档当前 head 状态
- 历史列表排序
- 保存接口中的乐观锁校验
- 回滚、Diff、审计日志追踪

## 4.3 保存 API 设计

### 查询文档

`GET /api/documents/{id}`

返回：

```json
{
  "id": 123,
  "title": "需求讨论纪要",
  "content": {},
  "headVersion": 58,
  "lastSavedAt": "2026-05-27T14:32:10Z"
}
```

### 自动/手动保存

`POST /api/documents/{id}/save`

请求体：

```json
{
  "baseVersion": 58,
  "content": {},
  "saveMode": "MANUAL",
  "summary": "完成第一版"
}
```

响应：

```json
{
  "revisionId": 901,
  "version": 59,
  "savedAt": "2026-05-27T14:35:00Z"
}
```

### 查询版本历史

`GET /api/documents/{id}/revisions?cursor=0&pageSize=20`

### 查询指定版本详情

`GET /api/documents/{id}/revisions/{revisionId}`

### 版本 Diff

`GET /api/documents/{id}/diff?fromVersion=55&toVersion=59`

### 回滚

`POST /api/documents/{id}/rollback`

请求体：

```json
{
  "targetVersion": 52,
  "baseVersion": 59,
  "reason": "回退到发布前版本"
}
```

响应：

```json
{
  "revisionId": 920,
  "version": 60,
  "rolledBackFrom": 52
}
```

## 4.4 如何用版本号实现乐观锁防止覆盖

保存时，前端必须携带 `baseVersion`，表示“我是基于哪个版本编辑的”。

服务端保存时执行以下逻辑：

1. 查询 `document.head_version`
2. 比较是否等于请求中的 `baseVersion`
3. 若相等，允许保存，并将版本号加 1
4. 若不相等，说明期间已有其他人提交新版本，返回冲突错误

伪代码如下：

```java
@Transactional
public SaveResult save(Long documentId, SaveDocumentRequest req, Long userId) {
    Document doc = documentMapper.selectById(documentId);
    if (!Objects.equals(doc.getHeadVersion(), req.getBaseVersion())) {
        throw new BizException("DOCUMENT_VERSION_CONFLICT");
    }

    long newVersion = doc.getHeadVersion() + 1;

    doc.setContentJson(req.getContent());
    doc.setHeadVersion(newVersion);
    doc.setLastSavedBy(userId);
    doc.setLastSavedAt(LocalDateTime.now());

    documentMapper.updateById(doc);

    DocumentRevision revision = new DocumentRevision();
    revision.setDocumentId(documentId);
    revision.setVersionNo(newVersion);
    revision.setSaveMode(req.getSaveMode());
    revision.setContentJson(req.getContent());
    revision.setSummary(req.getSummary());
    revision.setOperatorId(userId);
    revision.setCreatedAt(LocalDateTime.now());
    revisionMapper.insert(revision);

    return new SaveResult(revision.getId(), newVersion, doc.getLastSavedAt());
}
```

如果希望更严谨，可将更新写成带条件的 SQL：

```sql
UPDATE document
SET content_json = ?, head_version = head_version + 1, update_time = NOW()
WHERE id = ? AND head_version = ?;
```

然后检查影响行数：

- 影响 1 行：保存成功
- 影响 0 行：说明版本冲突

这就是标准的乐观锁实现方式。

## 4.5 版本历史是保存全量还是增量

建议采用“全量快照为主，增量 Diff 为辅”的折中方案。

### 方案对比

#### 仅保存全量快照

优点：

- 实现最简单
- 回滚最快
- Diff 可按任意两个版本现算

缺点：

- 存储占用更高

#### 仅保存增量操作

优点：

- 存储空间更省

缺点：

- 回放链条长，恢复复杂
- 容易因中间操作损坏导致还原失败
- 排查问题困难

### 推荐落地方式

- 每次正式版本保存完整快照到 `document_revision`
- 如有性能或存储压力，再额外保存 `diff_json`
- 超大文档可做“每 N 个版本一个快照，中间存增量”的混合模式

对当前阶段的业务来说，优先保证正确性和易维护性，建议先采用“每版全量快照”。

## 五、实时协作与版本系统如何配合

实时协作与版本系统分工不同：

- WebSocket/CRDT 负责“当前屏幕上内容怎么同步”
- 保存接口负责“什么时候形成一个可追溯版本”

推荐规则：

- 高频编辑过程通过实时协作同步，不频繁写历史版本
- 自动保存可以生成轻量版本，必要时做保留策略
- 手动保存一定生成清晰历史版本
- 回滚一定生成新版本

这样可以避免历史列表被无意义的高频编辑淹没。

## 六、关键异常场景处理

## 6.1 网络断开

- 前端进入 `offline` 状态
- 本地继续编辑并写入 IndexedDB
- 重连后先拉取最新 `headVersion`
- 若本地是基于旧版本修改，则尝试 CRDT 合并；若是普通文本模式，则提示用户对比后再提交

## 6.2 多人同时手动保存

- 通过 `baseVersion + headVersion` 乐观锁检查发现冲突
- 后提交者收到冲突提示
- 前端可提供“查看最新版本差异并重新提交”能力

## 6.3 正在回滚时有人编辑

- 回滚请求同样必须带 `baseVersion`
- 若 headVersion 已变化，则拒绝本次回滚并提示刷新
- 防止用户把别人的最新修改直接覆盖掉

## 6.4 页面异常关闭

- `beforeunload` 阶段尽量使用 `sendBeacon` 上报草稿
- 若来不及发送，则依赖 IndexedDB 本地恢复

## 七、落地建议

按实现优先级，建议分三期推进：

### 第一期：先做稳定版

- 单文档编辑器
- 5 秒自动保存 + 手动保存
- 保存状态展示
- 版本历史列表
- 版本回滚
- MySQL 全量快照存储
- 前端 IndexedDB 草稿恢复

### 第二期：增强协作

- WebSocket 实时协作
- 在线协作者状态显示
- 乐观锁冲突提示优化
- 版本 Diff 界面

### 第三期：提升大规模场景能力

- CRDT/Yjs 正式接入
- Redis 广播与多实例横向扩展
- 增量操作日志
- 大文档分块保存和快照压缩

## 八、结论

这套方案的核心思想是：

- 用实时协作机制解决“多人同时编辑”
- 用自动保存和本地草稿恢复解决“数据不丢”
- 用版本表和版本号解决“历史可追溯、可回滚”
- 用 `baseVersion` 乐观锁解决“保存覆盖”问题
- 用“回滚生成新版本”而不是直接覆盖旧版本，保证历史链完整

如果以当前项目技术栈落地，推荐的最小可用实现是：

- React 编辑器 + IndexedDB 草稿恢复
- Spring Boot 保存/历史/回滚 API
- MySQL 存当前文档 + 历史快照
- 保存接口基于 `head_version` 做乐观锁
- 后续再逐步补充 WebSocket 协作与 Diff 细化能力
